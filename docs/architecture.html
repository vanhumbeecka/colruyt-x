Architecture overview

Colruyt-X

A small sale-notification service for two people. You keep a shared watchlist of Colruyt grocery products; every day the system checks whether any watched product entered or improved a volume deal ("buy N, pay X each") and sends a Telegram message when it does. It is a full-stack TypeScript app — a React SPA and an Express API — deployed on Vercel and backed by a Turso (libSQL) database.

Components

Colruyt GCS bucket

Public Google Cloud Storage feed (colruyt-products) with a daily JSON snapshot of the ~15k-product catalogue. The source of truth for prices and volume deals.

React SPA

The user-facing app: PIN login, the shared watchlist, and product search. Built by Vite and served by Vercel as static files.

Express API

A single serverless function. Handles auth (HMAC-signed cookie / Bearer token), product search, watchlist CRUD, and the cron endpoint.

Cron handler

The daily pipeline: import the latest snapshot, run pure detectDeals against stored state, then notify. Gated by CRON_SECRET.

Turso (libSQL)

SQLite-compatible database. products mirrors the catalogue, watchlist_items is the shared list, and deal_state remembers what was already notified so a standing deal isn't re-sent daily.

Telegram Bot API

The notification channel. The notifier formats one message per deal event (with the product image when available) and posts to each configured chat.

The daily flow

  1. Vercel Cron calls GET /api/cron/import-products at 06:00, presenting CRON_SECRET as a Bearer token.
  2. Import — the handler finds the latest GCS snapshot, downloads it, and upserts every product into Turso. A failure here aborts the run so detection never sees stale data.
  3. Detect — it loads the watchlist plus the previous deal_state and runs the pure detectDeals function, emitting an event whenever a watched product newly enters a volume deal or its per-item price improves.
  4. Notify — each event is formatted and sent to Telegram. Only after notifications succeed is the new deal_state saved, so a Telegram failure simply retries next run.
  5. Meanwhile, the two users browse the React SPA, log in with a PIN, search products, and add/remove items on the shared watchlist — all through the Express API against the same Turso database.
